### 3. 미완성 코드 데이터 수집에 대한 핵심 질문
- 질문: 제안서에 명시된 "100명의 참가자를 통한 직접 수집"의 근본적인 목적이 학생들의 생산적 어려움(Productive Struggle) 과정을 추적하기 위함인지 명확히 하고자 함.

**Branch 3-1: 목적이 "생산적 어려움(과정)" 추적이 맞을 경우**
- 결과물(제출 코드)뿐만 아니라 소요 시간, 수정 횟수, 키스토로크 단위의 구체적인 행동 로그가 필수적임.
- 기존 데이터로는 불가능하므로, 당장 데이터를 모으기보다 이러한 세밀한 로그를 추적하고 기록할 수 있는 실험용 플랫폼 개발이 필요함. 
- 개발 방향
    - 타이핑이 1~2초 멈췄을 때, 코드의 변화를 묶어서 저장
    - 좀 오래동안 타이핑이 멈췄을 경우, 어떤 이유로 멈추게 되었는지 선택하게 하기 (스스로 풀이 고민, 구글링 등 코드 검색, 휴식, 기타)
    - onDidChangeTextDocument을 이용해서 지워진 내용도 기록할 수 있을 것으로 보임
        - onDidChangeTextDocument: trigger(키보드 입력) 때마다 코드의 변경 사항, 커서 위치 등을 측정해주는 API
    - ProgSnap2라는 데이터 표준을 따라 로그 저장.
- 장점
    - 추후 서비스를 VS code 기반으로 생성할 것이라면, 해당 경험이 실제 서비스를 만드는데 도움이 될 것으로 보임.
    - 기존에 존재하는 API를 이용할 수 있기 때문에, 효율적인 구현이 가능

**Branch 3-2: 목적이 단순히 "에러와 개념의 매핑(결과)"일 경우**
- 굳이 사람을 모아 수집하는 것은 시간/비용 대비 비효율적일 것 같음
- 기존에 구축된 프로그래밍 코드 데이터셋에 LLM을 이용해서 코드의 뒷부분을 잘라내거나(Truncation), 오류를 주입함으로써(Error Injection) 대규모 미완성 코드를 생성하는 방향을 제안
    - 이용할 수 있는 코드 데이터
        - MBPP (Mostly Basic Python Problems) - 기초 파이썬 문제셋
        - APPS (Automated Programming Progress Standard)
        - LeetCodeDataset (Hugging Face)
        - CodeSearchNet (GitHub & Weights & Biases)
- 신뢰도에 문제가 있다면, 소규모 (10~20명 정도) 인원으로 실제 데이터를 수집해서 함께 이용하는 방식도 있음
