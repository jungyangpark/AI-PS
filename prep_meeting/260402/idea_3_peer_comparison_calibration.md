# 아이디어 3: 동료 비교 기반 생산적 어려움 보정

## 요약

**익명화된 동료 데이터**를 활용하여 다양한 기술 수준에서 "생산적 어려움"이 어떤 모습인지를 보정함. 절대적 임계값(예: "3분 이상 막히면 도움 제공")만 사용하는 대신, 현재 학습자의 행동을 유사한 지식 상태를 가진 동료들의 분포와 비교하여, 어려움이 생산적인지 비생산적인지에 대해 더 맥락적으로 인식된 결정을 내림.

## 동기

AI-PS 프레임워크의 핵심 과제는 **언제 개입할지** 결정하는 것:
- 너무 이르면 → 학생의 생산적 어려움을 빼앗음
- 너무 늦으면 → 좌절감이 이탈과 비생산적 어려움으로 이어짐

과제 제안서는 시간 기반 및 편집 기반 휴리스틱을 사용하지만, 이에는 문제가 있음:
- **재귀에서 3분 막힘**은 초보자에게는 정상이지만 숙련자에게는 위험 신호일 수 있음
- **빠르게 타이핑하고 삭제함**은 탐색(좋음)일 수도 혼란(나쁨)일 수도 있음
- **개인 기준선이 크게 다름** — 일부 학생은 자연스럽게 더 느리게 작업함

동료 비교는 다음과 같은 질문으로 이를 해결함: **"이 학생의 행동이 유사한 수준에서 유사한 개념을 다루는 다른 학생들과 비교하여 어떠한가?"**

## 작동 방식

### 데이터 수집 단계 (1분기)
~100명의 참가자를 대상으로 한 초기 데이터셋 수집 중:
- 코드 스냅샷뿐만 아니라 전체 행동 트레이스 수집:
  - 개념별 소요 시간
  - 편집 패턴 (삽입, 삭제, 수정)
  - 정지 시간
  - 오류 빈도
  - 도움 요청 행동
- 결과 레이블링: 학생이 최종적으로 해당 개념을 숙달했는가? 얼마나 걸렸는가?

### 동료 프로필 구축
전체 성과를 기반으로 학생들을 **기술 수준별 코호트**로 클러스터링:
```
[초보자 코호트]   → "반복문"에 대한 일반적 어려움 시간: 5-8분
[중급자 코호트]   → "반복문"에 대한 일반적 어려움 시간: 2-4분
[고급자 코호트]   → "반복문"에 대한 일반적 어려움 시간: 0.5-1분
```

각 코호트에 대해 개념별 **행동 분포**를 계산:
- 완료까지의 중앙값 시간
- 일반적 오류 횟수
- 최종적으로 숙달에 이른 일반적 어려움 패턴 (생산적)
- 포기로 이어진 일반적 어려움 패턴 (비생산적)

### 실시간 보정
새 학생이 AI-PS 시스템을 사용할 때:
1. 시스템이 현재 기술 수준을 추정함 (학습자 모델에서)
2. 각 코딩 과제에 대해, 해당 기술 수준 + 개념에 대한 동료 행동 분포를 조회함
3. 학생의 현재 행동을 분포와 비교함:

```
"재귀"에 대한 학생의 시간: 6분
동료 분포 (중급 수준):
  - 25번째 백분위수: 2분 (빠른 해결자)
  - 50번째 백분위수: 4분 (일반적)
  - 75번째 백분위수: 7분 (느리지만 여전히 생산적)
  - 90번째 백분위수: 12분 (막힌 것으로 추정)

→ 학생은 ~70번째 백분위수 → 여전히 생산적 범위 내 → 아직 개입하지 않음
```

### 의사결정 프레임워크

| 동료 분포에서의 학생 위치 | 해석 | 행동 |
|---|---|---|
| 50번째 백분위수 이하 | 일반적보다 빨리 해결 중 | 현재 지원 수준 유지 |
| 50번째 - 75번째 백분위수 | 일반적인 생산적 어려움 | 현재 지원 수준 유지 |
| 75번째 - 90번째 백분위수 | 연장된 어려움, 아직 생산적일 가능성 | 미묘한 격려 표시 ("어려운 개념을 잘 파고들고 있네요!") |
| 90번째 백분위수 이상 | 비생산적 어려움으로 추정 | 폴백 트리거 (Parsons 문제 또는 1단계 지원) |
| 95번째 백분위수 이상 | 높은 좌절 위험 | 설명과 함께 즉각적 전체 지원 |

### 적응형 임계값
임계값 자체도 적응할 수 있음:
- 코호트 내 많은 학생이 특정 개념에서 어려움을 겪으면, 본질적으로 어려운 개념일 수 있음 → 시스템이 더 인내심을 가짐
- 특정 학생이 지속적으로 코호트보다 빠르거나/느리게 문제를 해결하면, 개인 보정을 조정

```
┌──────────────────────────────┐
│  현재 학생의 행동               │──→ [백분위수 계산기] ──→ [개입 결정]

```

## 프라이버시 및 윤리적 고려사항
- 모든 동료 데이터는 **완전히 익명화** 되어야 함 — 개인 식별 불가
- 학생들에게 행동 데이터가 수집됨을 고지해야 함 (IRB에서 다룸)
- 동료 데이터는 집계 형태로만 사용 (분포, 개별 트레이스가 아님)
- 데이터 사용을 원하지 않는 학생을 위한 **옵트아웃** 옵션 고려
- 학생들에게 절대로 "순위"를 보여주거나 경쟁 압력을 만들지 않음 — 비교는 시스템 내부에서만 사용

## 장점
1. **맥락 인식 임계값:** 자의적인 "3분 = 막힘" 규칙이 더 이상 필요 없음
2. **기술 수준에 맞는 기대:** 초보자에게는 더 많은 인내, 고급자에게는 더 빠른 개입
3. **개념 인식:** 시스템이 일부 개념이 본질적으로 더 어렵다는 것을 알고 있음
4. **자기 개선:** 더 많은 학생이 시스템을 사용할수록 동료 분포가 더 정확해짐
5. **거짓 양성 감소:** 자연스럽게 느리지만 생산적인 학습자에 대한 불필요한 개입 감소
6. **거짓 음성 감소:** 비정상적으로 막혀있는 고급 학생을 포착

## 데이터 요구사항
- **최소 실행 가능:** 3개 기술 수준에 걸친 ~100명의 학생 (1분기 데이터 수집이 이를 충족)
- **이상적:** 견고한 개념별 분포를 위해 500+명의 학생
- **콜드 스타트:** 동료 데이터가 없는 새로운 개념에 대해서는 시간 기반 휴리스틱으로 폴백
- 더 많은 학생이 시스템을 사용할수록 데이터셋이 성장하여 보정이 개선됨

## 잠재적 도전 과제
- **콜드 스타트 문제:** 초기에는 제한된 동료 데이터; 폴백 휴리스틱 필요
- **코호트 정의:** 학생들을 기술 수준별로 정확하게 클러스터링하는 방법
- **개념 세분화 수준:** 너무 세분화 → 데이터 희소; 너무 거친 → 부정확
- **문화적/맥락적 차이:** 동료 기준이 기관마다 다를 수 있음
- **시간적 변화:** 커리큘럼이 바뀌면 동료 기준선도 변할 수 있음

## 과제 제안서 타임라인과의 연관
- **1분기:** 전체 행동 트레이스를 캡처하는 데이터 수집 설계 (코드뿐만 아니라); 이것이 기반이 됨
- **2분기:** 학습자 모델의 일부로 동료 행동 데이터베이스와 백분위수 계산 파이프라인 구축
- **3분기:** 자동완성 결정 로직에 동료 보정 임계값 통합
- **4분기:** 동료 보정 임계값이 고정 임계값보다 우수한지 평가 (사용자 연구의 추가 분석이 될 수 있음)

## 가능한 확장: 동료 학습 넛지
보정 외에도, 동료 데이터를 통해 다음이 가능:
- "많은 학생들이 이 단계 전에 [개념 X]를 복습하는 것이 도움이 되었습니다"
- "이 개념을 숙달한 학생들은 보통 [특정 하위 문제]에 추가 시간을 투자했습니다"
- 이러한 넛지는 익명화되어 있으며 교육적이고, 경쟁적이지 않음
