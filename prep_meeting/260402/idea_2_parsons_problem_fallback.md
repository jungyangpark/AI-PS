# 아이디어 2: 중간 비계로서의 Parsons 문제 폴백

## 요약

학습자가 막혀있고 AI-PS 시스템이 자동완성을 제공하지 않는 상태(2단계: 내재화)에서, 전체 자동완성을 결국 보여주는 것(1단계로 폴백) 대신에, **Parsons 문제** — 학생이 올바른 순서로 재배열해야 하는 뒤섞인 코드 줄들의 집합 — 를 제공하는 것을 제안함. 이를 통해 처음부터 코드를 작성하는 것과 수동적으로 자동완성을 받아들이는 것 **사이의 중간 난이도 수준**을 만들 수 있음.

## 동기

현재 과제 제안서는 세 가지 단계를 기술함:
1. **초기 학습:** 자동완성을 보여주되 직접 타이핑하도록 요구
2. **내재화:** 자동완성을 제공하지 않음 (너무 오래 막히면 → 1단계로 폴백)
3. **숙달:** 전체 자동완성 제공

2단계 → 1단계 폴백의 문제점은 **이분법적 점프**라는 것: "직접 쓰세요"에서 "여기 답이 있습니다, 그냥 타이핑하세요"로 한 번에 넘어감. 이는 난이도가 점진적으로 감소하는 **단계적 비계** 접근의 기회를 놓치는 것임.

Parsons 문제는 다음 사이의 "적절한 지점"에 위치함:
- **너무 어려움:** 아무런 지원 없이 처음부터 코드 작성 (좌절적 어려움)
- **너무 쉬움:** 전체 자동완성을 보고 복사 (수동적, 학습 없음)

연구에 따르면 Parsons 문제는:
- 코드를 작성하는 것보다 시간이 덜 걸리면서도 학습에 동등하게 효과적임
- 코드 구조에 대한 오개념을 노출시키는 데 더 효과적임
- 직접 해답을 받는 것보다 학생들이 선호함

## 작동 방식

### 현재 흐름 (과제 제안서)
```
2단계에서 학생 막힘 ──(타임아웃)──→ 1단계로 폴백 (전체 완성 + 타이핑)
```

### 제안하는 흐름 (Parsons 폴백 포함)
```
2단계에서 학생 막힘 ──(타임아웃 1)──→ Parsons 문제 제공 (뒤섞인 코드 줄)
                                                │
                                      ┌─────────┴─────────┐
                                      │                    │
                                 학생이 해결함        여전히 막힘 (타임아웃 2)
                                (학습 발생!)                │
                                                           ▼
                                                 1단계로 폴백
                                                (전체 완성 제공)
```

### Parsons 문제 생성
1. 시스템은 이미 의도된 완성 코드를 알고 있음 (자동완성 엔진에서).
2. 완성 코드를 논리적 줄/블록으로 분리.
3. 순서를 뒤섞고 IDE에서 드래그 앤 드롭 퍼즐로 제시.
4. 선택적으로 **교란 항목** (잘못된 줄)을 추가하여 난이도를 높임.

LLM이 맥락에 적합한 Parsons 문제를 생성할 수 있음:
- 학생의 미완성 코드를 맥락으로 활용
- 올바른 완성 코드를 생성
- 줄로 분리하고 섞기
- 선택적으로 1-2개의 그럴듯하지만 잘못된 교란 줄 생성

### 예시 인터랙션

**학생의 미완성 코드:**
```python
def binary_search(arr, target):
    left, right = 0, len(arr) - 1
    # 학생이 여기서 막힘...
```

**전체 완성을 보여주는 대신, Parsons 문제를 제공:**

> binary_search 함수를 완성하기 위해 아래 줄들을 올바른 순서로 배열하세요:

```
[ ] return mid
[ ] right = mid - 1
[ ] while left <= right:
[ ] if arr[mid] == target:
[ ] left = mid + 1
[ ] elif arr[mid] < target:
[ ] mid = (left + right) // 2
[ ] return -1
```

학생은 이를 올바른 순서로 드래그하여 배열해야 하며, 처음부터 모든 세부 사항을 기억해내지 않아도 로직에 참여하게 됨.

## VS Code에서의 UI/UX 설계

### 옵션 A: 인라인 패널
- 현재 코드 아래에 뒤섞인 줄을 보여주는 작은 패널 등장
- 학생이 순서를 드래그하고, "제출"을 클릭하여 코드를 삽입
- 코딩 흐름에 최소한의 방해

### 옵션 B: 사이드 패널
- VS Code의 전용 사이드 패널에 Parsons 문제를 표시
- 더 긴 코드 블록과 교란 항목을 위한 더 많은 공간 제공
- 테스트 중인 개념에 대한 힌트를 포함할 수 있음

### 옵션 C: 빈칸이 있는 고스트 텍스트
- 완성 코드를 고스트 텍스트로 보여주되 일부 줄을 비워둠/숨김
- 학생이 빠진 부분을 채움 (하이브리드 접근)

## 난이도 조정

학습자 상태에 따라 Parsons 문제의 난이도를 조절할 수 있음:

| 학습자 상태 | Parsons 난이도 | 세부 내용 |
|---|---|---|
| 2단계 진입 직후 | 쉬움 | 배열할 줄 수가 적음, 교란 항목 없음, 논리적 그룹 힌트 |
| 2단계 중반 | 보통 | 더 많은 줄, 교란 항목 1-2개, 힌트 없음 |
| 폴백 직전 | 어려움 (하지만 빈 화면보다는 쉬움) | 많은 줄, 다수의 교란 항목 |

## 장점
1. **단계적 비계:** "도움 없음"과 "전체 답"의 간격을 메움
2. **능동적 참여:** 학생이 코드 구조와 로직에 대해 추론해야 함
3. **오개념 노출:** 잘못된 순서 배열이 특정 오해를 드러냄
4. **빠른 완료:** 처음부터 작성하는 것보다 시간이 적게 걸려, 몰입 상태 유지
5. **풍부한 데이터:** 배열 시도가 학습자 모델의 추가 신호를 제공
6. **연구 기반:** Parsons 문제는 CS 교육에서 강력한 실증적 뒷받침을 가짐

## 잠재적 도전 과제
- VS Code는 기본적으로 드래그 앤 드롭 UI를 지원하지 않음 — 커스텀 웹뷰 패널이 필요
- 좋은 Parsons 문제(특히 교란 항목) 생성을 위해 신중한 프롬프트 엔지니어링 필요
- 설계가 신중하지 않으면 코딩 흐름을 방해할 수 있음
- 학생들이 실제로 도움이 된다고 느끼는지 vs. 성가시다고 느끼는지 평가 필요
